## 3. 什么是 Prompt？

Prompt 可以理解为：

> 用户给大语言模型的任务说明书。

比如下面这句话就是一个 Prompt：

```text
解释一下机器学习。
```

但是这个 Prompt 很简单，模型并不知道你希望它从哪个角度解释。

它不知道你想要：

```text
简单解释？
数学解释？
适合初学者？
适合面试回答？
适合写进论文？
要不要举例？
要不要输出成 Markdown？
```

所以更好的 Prompt 是：

```text
请你用适合研一数学专业学生理解的方式，解释什么是机器学习。

要求：
1. 先给一句话定义。
2. 再用生活例子解释。
3. 再说明机器学习和传统编程的区别。
4. 最后用 Markdown 表格总结。
```

可以看到，Prompt 不是简单地“问一句话”，而是把任务说明清楚。

---

## 4. Prompt 的本质

Prompt 的本质不是“咒语”，而是：

```text
把任务说清楚
把背景说清楚
把要求说清楚
把输出格式说清楚
```

大语言模型并不是天然知道用户想要什么。

如果 Prompt 太模糊，模型就只能猜。

如果 Prompt 更清楚，模型就更容易输出符合需求的结果。

---

## 5. 普通提问 vs 结构化 Prompt

### 5.1 普通提问

```text
讲讲 Transformer。
```

这个问题太宽泛。

模型可能不知道应该：

```text
讲直觉？
讲公式？
讲 Encoder？
讲 Decoder？
讲 Self-Attention？
讲代码实现？
讲面试回答？
```

---

### 5.2 更好的 Prompt

```text
请你面向刚学完 Attention 的机器学习初学者，解释 Transformer 的基本结构。

要求：
1. 不要一上来堆公式。
2. 先解释 Encoder 和 Decoder。
3. 再解释 Self-Attention 的作用。
4. 最后说明 Transformer 为什么比 RNN 更适合处理长文本。
5. 用 Markdown 小标题组织内容。
```

这个 Prompt 比普通提问更清楚。

它告诉模型：

```text
回答对象是谁
要解释什么
解释到什么程度
不要怎么讲
最后输出成什么形式
```

## 6. 一个好 Prompt 的基本结构

一个比较通用的 Prompt 结构是：

```text
角色 + 任务 + 背景 + 要求 + 输出格式
```

也可以写成：

```text
你是谁？
你要做什么？
背景是什么？
有什么限制？
输出成什么样？
```

这五个部分分别是：

| 组成部分          | 含义      | 示例                          |
| ------------- | ------- | --------------------------- |
| Role          | 角色      | 你是一名机器学习老师                  |
| Task          | 任务      | 请解释什么是 Prompt Engineering   |
| Context       | 背景      | 我刚学完 Transformer 和 GPT/BERT |
| Constraints   | 要求 / 限制 | 不要堆公式，每个概念都要举例              |
| Output Format | 输出格式    | 按照定义、直觉、例子、总结输出             |

---

## 7. Prompt 五要素详解

### 7.1 Role：角色

Role 用来告诉模型应该以什么身份回答。

例如：

```text
你是一名机器学习老师。
```

或者：

```text
你是一名 AI Agent 工程师。
```

角色的作用是帮助模型选择合适的知识范围、表达方式和回答风格。

---

### 7.2 Task：任务

Task 用来告诉模型要完成什么事情。

例如：

```text
请解释什么是 Prompt Engineering。
```

或者：

```text
请帮我把下面这段内容整理成 GitHub 学习笔记。
```

任务要尽量明确。

不好的任务：

```text
帮我看看这个。
```

更好的任务：

```text
请你检查下面这段 Python 代码是否有 bug，并解释错误原因。
```

---

### 7.3 Context：背景

Context 用来告诉模型当前情况。

例如：

```text
我刚学完 Transformer 和 GPT/BERT 基础，现在准备学习 LLM 应用。
```

背景越清楚，模型越容易给出贴合当前阶段的回答。

如果没有背景，模型可能会讲得太浅，也可能讲得太深。

---

### 7.4 Constraints：要求和限制

Constraints 用来控制回答的难度、风格、长度和重点。

例如：

```text
要求：
1. 不要使用太多专业术语。
2. 每个概念都要举一个例子。
3. 输出内容适合放进 Markdown 学习笔记。
4. 不要写成聊天记录。
```

限制条件可以让模型的输出更符合预期。

---

### 7.5 Output Format：输出格式

Output Format 用来规定答案应该长什么样。

例如：

```text
请按照以下格式输出：

1. 概念定义
2. 直觉理解
3. 例子
4. 常见误区
5. 总结表格
```

输出格式非常重要。

很多时候，模型回答得乱，不是因为模型不会，而是因为没有告诉它应该如何组织答案。

## 8. 标准 Prompt 模板

以后可以经常使用下面这个模板：

```text
你是一名【角色】。

请你帮我完成【任务】。

背景：
【说明当前情况、学习阶段、已有基础】

要求：
1. 【要求一】
2. 【要求二】
3. 【要求三】

输出格式：
1. 【第一部分】
2. 【第二部分】
3. 【第三部分】
```

例如：

```text
你是一名机器学习老师。

请你面向初学者解释什么是 Transformer。

背景：
我已经学过 Embedding 和 Attention，但还不太理解 Encoder、Decoder 和 Self-Attention 的关系。

要求：
1. 不要一上来堆公式。
2. 先讲直觉，再讲结构。
3. 每个核心概念都要举例。
4. 最后给出总结表格。

输出格式：
1. Transformer 是什么
2. 为什么需要 Transformer
3. Encoder 和 Decoder
4. Self-Attention 的作用
5. 本节总结
```

## 14. Zero-shot Prompt

Zero-shot Prompt 指的是：

> 不给示例，直接让模型完成任务。

例如：

```text
请判断下面这句话的情感是正向、负向还是中性：

这家店味道不错，下次还来。
```

这里没有给模型任何示例，所以是 Zero-shot。

---

## 15. One-shot Prompt

One-shot Prompt 指的是：

> 给 1 个示例，然后让模型模仿这个示例完成任务。

例如：

```text
请判断句子的情感是正向、负向还是中性。

示例：
句子：这个电影太好看了。
情感：正向

现在判断：
句子：这家店味道不错，下次还来。
情感：
```

这里给了一个示例，所以是 One-shot。

---

## 16. Few-shot Prompt

Few-shot Prompt 指的是：

> 给少量示例，让模型模仿示例完成任务。

例如：

```text
请判断下面句子的情感是正向、负向还是中性。

示例 1：
句子：这个电影太好看了。
情感：正向

示例 2：
句子：这个服务真的很差。
情感：负向

示例 3：
句子：今天是星期三。
情感：中性

现在判断：
句子：这家店味道不错，下次还来。
情感：
```

前面的几个示例会帮助模型理解任务格式和判断标准。

---

## 17. Zero-shot、One-shot、Few-shot 对比

| 类型        | 含义           | 示例数量   |
| --------- | ------------ | ------ |
| Zero-shot | 不给示例，直接做任务   | 0 个    |
| One-shot  | 给 1 个示例，再做任务 | 1 个    |
| Few-shot  | 给少量示例，再做任务   | 2 个或多个 |

可以这样记：

```text
Zero-shot：直接做
One-shot：看一个例子再做
Few-shot：看几个例子再做
```

## 18. Chain-of-Thought Prompt

Chain-of-Thought，简称 CoT，通常翻译为：

```text
思维链提示
```

它的核心思想是：

> 引导模型先分解问题，再给出答案。

例如普通提问：

```text
小明有 3 个苹果，小红给了他 2 个，他吃掉 1 个，还剩几个？
```

更好的写法：

```text
请你先列出解题步骤，再给出最终答案：

小明有 3 个苹果，小红给了他 2 个，他吃掉 1 个，还剩几个？
```

对于复杂任务，分步骤分析通常比直接给答案更稳定。

不过需要注意：

```text
简单任务不一定需要 CoT。
复杂任务可以要求模型给出简洁步骤、判断依据或解题过程。
```

## 19. Prompt 常见错误

### 19.1 错误一：任务太模糊

不好的 Prompt：

```text
帮我看看这个。
```

问题是模型不知道“看看”是什么意思。

更好的 Prompt：

```text
请你检查下面这段代码是否存在 bug，并解释错误原因。
```

---

### 19.2 错误二：没有背景

不好的 Prompt：

```text
帮我规划学习路线。
```

问题是模型不知道你的基础、目标和时间安排。

更好的 Prompt：

```text
我是研一数学专业学生，已经学过传统机器学习和 PyTorch 基础，现在想学习 AI Agent。请你给我规划一条从 LLM 到 RAG 再到 Agent 的学习路线。
```

---

### 19.3 错误三：没有输出格式

不好的 Prompt：

```text
总结一下这篇文章。
```

更好的 Prompt：

```text
请按照以下格式总结：

1. 研究背景
2. 核心问题
3. 方法
4. 实验结果
5. 可以借鉴的地方
```

---

### 19.4 错误四：要求互相冲突

不好的 Prompt：

```text
帮我写得专业、简单、详细、短一点、全面一点。
```

这里的要求可能互相冲突。

更好的 Prompt：

```text
优先保证初学者能看懂，其次保证结构完整，不追求特别短。
```

## 20. Prompt 和 RAG 的关系

RAG 的全称是：

```text
Retrieval-Augmented Generation
检索增强生成
```

RAG 的基本流程是：

```text
用户问题
↓
检索相关文档
↓
把文档内容放进 Prompt
↓
让大语言模型基于文档回答
```

也就是说，RAG 的核心之一就是：

```text
把检索到的外部知识组织进 Prompt
```

没有 RAG 时，模型主要依赖自己参数中的知识。

有 RAG 时，模型可以参考外部文档回答问题。

所以 RAG 可以理解为：

```text
Prompt + 外部知识库
```

## 21. Prompt 和 Agent 的关系

Agent 可以理解为：

> 能够根据任务进行规划，并调用工具完成任务的大模型应用。

Agent 的基本流程是：

```text
用户任务
↓
Prompt 告诉模型它的角色、目标和工具
↓
模型决定是否调用工具
↓
工具返回结果
↓
模型根据结果继续行动或回答
```

在 Agent 里面，Prompt 会告诉模型：

```text
你是谁
你要完成什么任务
你有哪些工具
什么时候应该调用工具
工具结果应该如何使用
最终结果应该如何输出
```

所以 Agent 也离不开 Prompt。

可以简单理解为：

```text
Agent = LLM + Prompt + Tools + Memory + Planning
```

Prompt 是 Agent 的基础组成部分之一。

## 22. Prompt 的实用检查清单

写 Prompt 时，可以用下面这个清单检查：

```text
1. 我有没有说明角色？
2. 我有没有说清楚任务？
3. 我有没有提供必要背景？
4. 我有没有说明限制条件？
5. 我有没有规定输出格式？
6. 我有没有提供示例？
7. 我有没有说明判断标准？
8. 我有没有说明回答对象是谁？
```

如果一个 Prompt 效果不好，可以优先检查：

```text
任务是否太模糊？
背景是否太少？
格式是否没规定？
要求是否冲突？
示例是否不足？
```

## 24. 本节重点概念整理

| 概念                 | 含义                        |
| ------------------ | ------------------------- |
| Prompt             | 给大模型的任务说明书                |
| Prompt Engineering | 设计和优化提示词的方法               |
| Role               | 角色，告诉模型以什么身份回答            |
| Task               | 任务，告诉模型要做什么               |
| Context            | 背景，告诉模型当前情况               |
| Constraints        | 要求或限制，用于控制回答              |
| Output Format      | 输出格式，规定答案结构               |
| Zero-shot          | 不给示例，直接让模型完成任务            |
| One-shot           | 给一个示例，再让模型完成任务            |
| Few-shot           | 给多个示例，让模型模仿完成任务           |
| CoT                | 引导模型分步骤分析复杂问题             |
| RAG                | 把外部知识检索后加入 Prompt         |
| Agent              | 基于 Prompt、工具、记忆和规划完成任务的系统 |

## 25. 记忆口诀

可以用下面几句话记住 Prompt：

```text
Prompt 不是咒语，而是任务说明书。
任务越清楚，输出越稳定。
背景越充分，回答越贴合。
格式越明确，结果越好用。
```

再压缩成一句：

```text
Prompt = 角色 + 任务 + 背景 + 要求 + 输出格式
```

## 27. 本节总结

本节学习了 Prompt 的基础概念和常见写法。

核心结论是：

```text
Prompt 是给大语言模型的任务说明书。
好的 Prompt 可以让模型更准确地理解任务。
一个好 Prompt 通常包含角色、任务、背景、要求和输出格式。
Few-shot Prompt 可以通过示例让模型模仿任务。
CoT Prompt 可以帮助模型处理复杂推理任务。
RAG 和 Agent 都离不开 Prompt。
```

从整体学习路线看，Prompt 是从模型原理走向 LLM 应用的重要入口。

后续学习 RAG 和 Agent 时，会经常用到 Prompt。
